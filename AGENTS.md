# Repository Guidelines

## Project Purpose

이 repository는 2026년 10월 파리 + 코펜하겐 여행을 실제 현장에서 확인하기 위한 정적 여행 플래너입니다. 범용적인 `Travel Planner` UI를 유지하되, 현재 일정은 두 사람의 신혼여행 운영 계획이라는 맥락을 존중합니다.

핵심 목표는 휴대폰에서 날짜별 일정, 이동 동선, 체크리스트, 지도, 메모를 한 화면 흐름으로 빠르게 확인하는 것입니다. 랜딩 페이지나 홍보 사이트처럼 만들지 마세요.

## Source Of Truth

- 앱의 중심 파일은 `index.html`입니다. HTML, CSS, JavaScript를 한 파일에서 관리합니다.
- 여행 일정 데이터는 `index.html`의 `days` 배열에서 수정합니다.
- My Maps 링크는 스크립트 상단의 `MY_MAPS`, `MY_MAPS_EMBED` 상수에서 관리합니다.
- `TRAVEL_SAMPLE_HANDOFF.md`와 외부 `TRAVEL_LEGACY.md`는 작업 배경을 이해하기 위한 로컬 기록입니다. 앱 데이터와 충돌하면 사용자에게 확인하고 임의로 확정 정보를 바꾸지 마세요.
- 별도 요청이 없는 한 프레임워크, 백엔드, 패키지 매니저, 빌드 단계를 추가하지 마세요.

## Product Principles

- 모바일 사용성을 최우선으로 합니다.
- 날짜를 선택하면 시간표, 지도, 체크리스트, 메모를 같은 상세 화면에서 확인할 수 있어야 합니다.
- 한 날짜에는 보통 핵심 일정 2~3개와 식사, 이동, 휴식 여유를 둡니다.
- 매일 한 번은 산책, 전망, 좋은 식사처럼 신혼여행다운 시간을 남깁니다.
- 지도만 크게 노출하거나 여러 페이지를 오가게 하는 구조는 피합니다.
- Google Directions iframe은 사용하지 않습니다. My Maps embed와 외부 Google Maps 길찾기 링크 조합을 유지합니다.
- 사용자 메모와 체크 상태는 `localStorage`에만 저장하며, 키 변경 시 기존 데이터 호환성을 고려합니다.

## Travel Planning Rules

- 확정 항공편과 도시 순서는 사용자가 요청하지 않는 한 변경하지 않습니다.
- 확정 정보, AI 추천, 최신 확인이 필요한 항목을 명확히 구분합니다.
- 운영시간, 휴무일, 입장료, 예약 오픈 시점, 패스 포함 여부는 추측하지 않습니다.
- 최신성이 필요한 정보는 공식 관광지, 교통, 식당, 아울렛 사이트를 우선 확인하고 확인일을 기록합니다.
- 루브르, 베르사유, 생트샤펠, 인기 식당, 특별전 등 예약 위험이 큰 항목은 눈에 띄게 표시합니다.
- Paris Museum Pass는 현재 10/08 베르사유, 10/09 파리 박물관의 2일권 계획을 기준으로 하며, 시간 예약 필요 여부와 경제성을 별도로 확인합니다.
- 주요 장소에는 가능한 경우 근처 식사/카페 후보와 대체안을 함께 둡니다.

## Privacy And Publishing

- 예약번호, 여권 정보, 연락처, 개인 이메일, 비공개 문서 링크 같은 민감정보를 `index.html`, README, 커밋, 공개 배포물에 넣지 마세요.
- `TRAVEL_SAMPLE_HANDOFF.md`는 로컬 전용이며 `.gitignore` 상태를 유지합니다.
- 공개 배포 전 `rg`로 예약번호와 비공개 링크가 포함되지 않았는지 확인합니다.

## Editing And Verification

- 기존 구조와 스타일을 따르는 작고 직접적인 수정을 우선합니다.
- 사용자 노출 문구는 별도 요청이 없으면 한국어로 작성합니다.
- 날짜, 요일, 시간대, 도시 이동일, 공항 이동 여유가 서로 맞는지 확인합니다.
- 로컬 확인은 `python3 -m http.server 8080` 실행 후 `http://localhost:8080/`에서 합니다.
- UI 변경 후 데스크톱과 모바일 너비에서 날짜 선택, 도시 필터, 검색, 지도 탭, 체크리스트, 메모 저장을 확인합니다.
- 콘솔 오류, 가로 스크롤, 텍스트 겹침, 빈 지도 영역이 없어야 합니다.

## Deployment

- 정적 파일은 repository root에서 제공되어야 합니다.
- GitHub Pages를 사용할 경우 `main` branch의 `/root`를 배포 대상으로 합니다.
- repository 공개 전환이나 외부 배포처럼 공개 범위를 바꾸는 작업은 사용자 확인 후 진행합니다.
